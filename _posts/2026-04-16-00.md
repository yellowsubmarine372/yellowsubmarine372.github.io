## Session 1: The Context Graph — Decisions, Agents, and Governance

### From Full-Stack Applications to Decision Infrastructure

The conference opened with a vision for what graph-native AI applications actually look like in production. The core premise: customers need more than documentation — they need working examples. Ideally, with a single command, you could scaffold an entire full-stack architecture built on a graph foundation.

But the deeper question is more fundamental: **how do we actually make decisions in an organization?**

Every decision eventually materializes as structured data. A graph is the natural substrate to capture that — real-world data, relationships, and the full-stack application context all in one place.

### Logging Is Not Context

One of the sharpest observations of the session: **logging is just events — logs don't understand context.** There's no feedback loop. You can see *what* happened, but not *why*, and you can't trace the reasoning that led to a decision.

**Traceability of context is the core problem.** And the context graph is the proposed solution.

The real challenge in building a context graph is knowing *where* to instrument decisions. You have to consciously place the points at which decisions get recorded. This is simultaneously a UX problem and an infrastructure problem — and it feeds directly into governance.

### Why Agents Need Governance

Agents are remarkably good at solving problems. Given access to all your GitHub repos, ticketing systems, documentation, and tooling, an agent can act across resources in a way no single human can. **There is no human equivalent with that breadth of accessibility.** That's the power — and the risk.

Because agents can act everywhere, they need to be governed. The context graph is the governance layer. It captures not just what agents did, but the reasoning and decisions behind those actions.

## Session 2: The GraphRAG System That Asks Back — Intelligent Probing for Better Answers

![alt_text](/assets/posts/2026_04_16/session1.png)

### The Ambiguity Problem

Standard RAG pipelines have a quiet failure mode: **ambiguous queries return confidently wrong answers.**

Take a simple example — a user asks: *"the meeting is slow."*

What do they mean?
- Slow to start?
- Slow to join?
- Laggy screen sharing?
- Audio/video delay?

Each interpretation leads to completely different retrieval paths and answers. The system has no way to know which one is relevant.

### Three Naive Approaches — and Why They Fail

**(1) Top-K Retrieval for Disambiguation**

Retrieve the top-K documents and use them to generate a clarifying question. The problem: the document needed for disambiguation might be ranked K+1 — just outside the retrieval window. The right context never makes it in.

**(2) Generate Clarifying Questions from Retrieved Chunks**

Send 200 retrieved chunks to an LLM and ask it to generate one clarifying question. This is token-wasteful and noisy. You're paying a large context cost just to ask a single question.

**(3) Metadata Filtering**

Use metadata tags to filter and guide the user toward the right interpretation. This is too rigid — metadata can't capture deep, implicit, or emergent relationships between concepts.

The root insight: **we don't need more data. We need deeper relationships.**

### The Solution: Knowledge Graph as Clarification Engine

The proposed architecture replaces flat retrieval with a multi-agent system backed by a knowledge graph:

| Agent | Role |
|---|---|
| **Detective** | Assesses query ambiguity |
| **Strategist** | Hypothesizes on the knowledge graph |
| **Interviewer** | Asks probing clarifying questions |
| **Expert** | Generates the high-quality final answer |

Instead of trying to answer immediately, the system first determines *whether it should ask a question*, and if so, *which question would most reduce ambiguity* — using the structure of the knowledge graph to reason about what's unknown.

**RAG isn't a monologue problem.** It's a dialogue problem.

### Key Takeaways

1. **Architect for Dialogue** — Move beyond single-shot RAG to a "clarification loop."
2. **Give Your Agent a World Model** — Use a Knowledge Graph for robust reasoning, not just document retrieval.
3. **Ask with Purpose** — Design agents that autonomously decide *when* and *how* to ask, not agents that always ask or never ask.
4. **Measure What Matters** — Evaluate based on task success and user efficiency, not retrieval precision alone.

## Session 3: Context Engineering

![alt_text](/assets/posts/2026_04_16/session2.png)

### Bigger Context ≠ Better Context

There's a common misconception that more context always helps. It doesn't. **Bigger context windows don't automatically produce better results** — what matters is whether the *right* context is present, structured well, and relevant to the task.

**Prompt engineering** is a one-shot, static discipline: craft a clever phrase and hope. **Context engineering** is different — it's a dynamic systems discipline:

- Goal-driven
- Selectively curated for relevancy
- Structured for LLM consumption
- Evolves across the lifetime of an agent interaction

> Context engineering = building dynamic systems that provide the right information and tools in the right format such that an LLM can plausibly accomplish the task.

### Sources of Context

Context doesn't come from one place. It has four categories:

| Source | Examples |
|---|---|
| **Human** | User query, system prompt, dynamic prompts |
| **Cognitive** | Metadata, guardrails, policies, chain-of-thought |
| **Memory** | Working memory (attention/CoT), long-term (episodic, procedural, graph) |
| **Data / Tools** | Tool definitions, tool results, vector RAG, GraphRAG |

### How to Structure Context

Structure isn't just for humans — it matters for LLMs too:

- Group related information together
- Label clearly with headers or XML tags
- Build an information hierarchy
- Be concise and relevant — remove duplication
- Summarize key essence at the end (for attention mechanisms)
- These principles apply to tool outputs as much as to prompts

### Managing Context: Three Strategies

**(1) Offload / Store**
- Filesystem: notes, progress, tasks, long-term memory
- Vector database: episodic memory
- Graph: structural / procedural memory

**(2) Reduce / Compress**
- Summarize (with acknowledged information loss)
- Extract key pieces, offload full content
- Prune at agent and tool boundaries
- Attach to or enrich existing memory structures

**(3) Retrieve / Cache**
- RAG and GraphRAG retrieval patterns
- Optimal tool retrieval (descriptions + dependencies)
- Reconstructive memory: assemble the right content from different places
- Context caching via the context pyramid

### The Context Pyramid

A useful mental model for layering context by persistence:

```
        ┌─────────────────────┐
        │   EPHEMERAL (top)   │  tool outputs, reasoning, user query
        ├─────────────────────┤
        │   DYNAMIC (middle)  │  examples, memory, metadata, tool list
        ├─────────────────────┤
        │  PERSISTENT (base)  │  system prompt, role, policies, core facts
        └─────────────────────┘
```

The base is stable. The top changes every turn. Good context engineering means knowing which layer each piece of information belongs to.

## Closing: Evolving from RAG to GraphRAG

Across all three sessions, a consistent theme emerged: **the knowledge graph is the missing layer** in most AI systems.

A Neo4j knowledge graph combined with LLMs offers concrete improvements over vector-only RAG:

| Dimension | Improvement |
|---|---|
| **Relevancy** | More relevant answers via relationship-aware retrieval |
| **Context** | Domain-specific, structured, factual knowledge included |
| **Explainability** | Reasoning paths are traceable — users can see *why* |
| **Security** | Role-based access control on graph data |

The trajectory is clear: from static retrieval to dynamic context graphs, from monologue pipelines to dialogue systems, from logging events to capturing decisions. The graph isn't just a database — it's the substrate for building agents that can reason, remember, and explain themselves.
